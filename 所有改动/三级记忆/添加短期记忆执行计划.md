# 添加短期记忆（当前对话）执行计划

## 1. 背景分析

### 1.1 当前架构问题
当前 `ReactAgent` 类实现存在以下记忆缺陷：
- **无状态设计**：每次 `execute_stream()` 调用仅传递当前用户消息，不包含历史对话
- **重复计算**：相同查询触发重复工具调用（如多次定位、天气查询）
- **上下文断裂**：多轮对话无法维持连贯性，Agent 无法引用先前讨论内容

### 1.2 技术现状
- **框架**：基于 LangChain ReAct Agent 框架
- **Agent创建**：使用 `create_agent()` 函数，配置模型、提示词、工具和中间件
- **执行流程**：`execute_stream()` 构造仅含当前消息的 `input_dict` 并流式执行

## 2. 目标

### 2.1 核心目标
为 Agent 添加短期记忆能力，实现：
1. **对话历史保持**：存储完整会话消息序列
2. **上下文连贯**：支持多轮对话中的指代和延续
3. **减少重复**：避免相同信息的重复查询

### 2.2 非目标
- 长期知识存储（由向量库负责）
- 跨会话记忆持久化
- 用户个性化记忆

## 3. 技术方案

### 3.1 方案选择：LangChain 内存组件集成

**推荐方案**：`ConversationBufferMemory`
- **原理**：完整存储对话消息序列
- **优势**：
  - 简单可靠，无信息损失
  - 与 LangChain Agent 原生兼容
  - 支持流式执行模式
- **存储方式**：内存字典，按会话ID索引

**备选方案**：`ConversationBufferWindowMemory`
- **原理**：仅保留最近 N 条消息（可配置窗口大小）
- **适用场景**：需要控制上下文长度的应用

### 3.2 架构设计

#### 3.2.1 记忆存储结构
```python
{
    "session_id_1": ConversationBufferMemory(),
    "session_id_2": ConversationBufferMemory(),
    # ...
}
```

#### 3.2.2 会话管理
- **会话标识**：支持外部传入 `session_id` 或自动生成
- **记忆生命周期**：会话期间保持，会话结束可清理
- **并发安全**：支持多会话并行访问

#### 3.2.3 集成方式
**方案A**：修改 `create_agent` 调用，通过 `memory` 参数集成
```python
# LangChain 可能支持的记忆集成方式
agent = create_react_agent(
    model=chat_model,
    tools=tools,
    memory=memory_instance,  # 新增记忆参数
    # ...
)
```

**方案B**：在 `execute_stream` 中手动管理消息历史
```python
# 从 memory 加载历史消息
history_messages = memory.load_memory_variables({})["history"]
# 构造包含历史的 input_dict
input_dict = {"messages": history_messages + [current_message]}
```

### 3.3 配置设计

#### 3.3.1 配置文件扩展
在 `config/agent.yml` 中添加记忆配置：
```yaml
memory:
  type: "buffer"  # buffer 或 buffer_window
  max_tokens: 2000  # 最大token限制（可选）
  window_size: 10   # 窗口模式下的消息数量
  session_ttl: 3600 # 会话过期时间（秒）
```

#### 3.3.2 环境变量
- `MEMORY_TYPE`：记忆类型选择
- `MAX_MEMORY_TOKENS`：记忆token限制

## 5. 风险与应对

### 5.1 技术风险

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| LangChain API 不直接支持记忆集成 | 中 | 高 | 采用方案B（手动消息历史管理） |
| 流式执行与记忆保存时机冲突 | 低 | 中 | 在流式完成后异步保存记忆 |
| 多会话记忆内存泄漏 | 中 | 中 | 实现会话TTL和定期清理 |
| 上下文token超限 | 高 | 高 | 添加token计数和警告机制 |

### 5.2 兼容性风险
- **现有调用方适配**：需要更新所有调用 `execute_stream` 的代码
- **应对**：提供向后兼容模式（默认生成session_id）

### 5.3 性能风险
- **记忆增长导致延迟**：长对话历史增加提示词长度
- **应对**：配置记忆窗口限制，添加摘要压缩选项
