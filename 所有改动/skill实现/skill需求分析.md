# Skill 系统需求分析（简化版）

## 🎯 背景与问题

### 当前状态
现有项目中，工具通过 `@tool` 装饰器定义，如：
```python
@tool(description="获取指定城市的天气，以消息字符串的形式返回")
def get_weather(city: str) -> str: ...
```

### 核心问题
1. **描述过于简单**：仅说明功能，未说明使用场景、调用时机、参数格式
2. **缺乏分类引导**：LLM 无法快速判断工具类别
3. **缺少调用示例**：LLM 难以准确匹配用户意图
4. **参数说明不清晰**：容易导致参数传递错误

### 导致后果
- **LLM 工具选择犹豫**：增加思考步骤，消耗更多 Token
- **试错调用增多**：错误选择工具导致重新尝试
- **参数格式错误**：无效 API 调用浪费资源和时间

## 🎯 设计目标

### 主要目标
通过 **增强 Skill 描述质量**，帮助 LLM 更准确、快速地选择工具，减少试错，从而节约 Token。

### 具体指标
1. **工具选择准确率提升**：+30-50%
2. **平均思考步骤减少**：-30% Token 消耗
3. **参数验证失败减少**：-50% 错误调用
4. **保持代码简单**：最小化架构变更

## 🛠️ 解决方案（简化）

### 核心概念：EnhancedSkill
在现有 `@tool` 函数基础上，增加**结构化描述信息**，不改变原有调用逻辑。

```python
class EnhancedSkill:
    """增强型 Skill 描述类"""

    # 核心字段
    name: str                    # 工具名称（与 @tool 函数名一致）
    description: str             # 详细功能描述（何时调用、解决什么问题）
    category: str                # 分类：查询/计算/数据/通信
    examples: List[str]          # 2-3个调用示例（自然语言格式）
    parameters: Dict[str, str]   # 参数说明 {参数名: 说明}
    constraints: List[str]       # 使用限制（如"仅支持中国城市"）

    # 原有函数引用
    func: Callable               # 原有的 @tool 装饰函数

    def to_langchain_tool(self) -> Tool:
        """转换为 LangChain Tool，生成优化后的描述"""
        enhanced_description = f"""
{self.description}

分类: {self.category}
参数: {self._format_parameters()}
示例: {self._format_examples()}
限制: {self._format_constraints()}
        """.strip()

        return Tool(
            name=self.name,
            description=enhanced_description,
            func=self.func
        )
```

### 关键改进点

#### 1. **天气查询 Skill 优化对比**
```
❌ 原描述: "获取指定城市的天气，以消息字符串的形式返回"

✅ 优化后:
功能: 查询中国城市实时天气信息，包括温度、湿度、风力等
分类: 查询类
参数: city (必填) - 中文城市名称，如"北京"、"上海市"
示例:
  - "查询北京的天气"
  - "上海今天天气怎么样"
  - "get_weather(city='广州')"
限制:
  - 仅支持中国城市
  - 需要网络连接
  - 数据来自高德天气API
```

#### 2. **定位 Skill 优化**
```
❌ 原描述: "获取用户所在城市的名称，以纯字符串形式返回"

✅ 优化后:
功能: 根据用户IP地址自动定位所在城市
分类: 查询类
参数: 无
示例:
  - "我在哪个城市？"
  - "获取我的位置"
  - "定位用户所在城市"
限制:
  - 依赖用户公网IP
  - 定位精度为城市级别
```

## 📋 实施计划（简化版）

1. **分析现有工具**：列出所有 `@tool` 函数，识别描述不足
2. **设计 EnhancedSkill 类**：定义字段结构，保持最小化
3. **制定优化方案**：为每个工具编写增强描述草稿

### 阶段二：实现核心（1天）
1. **创建 EnhancedSkill 类**：`agent/enhanced_skill.py`
2. **重写关键工具**（选择3个典型）：
   - `WeatherEnhancedSkill`：天气查询
   - `LocationEnhancedSkill`：用户定位
   - `RagEnhancedSkill`：知识库检索
3. **实现转换方法**：`to_langchain_tool()` 生成优化描述

### 阶段三：集成测试（0.5天）
1. **修改 ReactAgent**：替换工具列表为 EnhancedSkill 转换结果
2. **功能验证**：确保原有功能正常工作
3. **简单效果测试**：对比优化前后 LLM 工具选择表现

## 🔧 技术要点

### 保持简单原则
1. **不改变原有调用逻辑**：EnhancedSkill 仅包装现有函数
2. **最小架构变更**：不引入复杂注册、发现机制
3. **向后兼容**：随时可回退到原始 @tool 描述
4. **逐步实施**：先优化关键工具，验证效果后再扩展

### 与现有代码关系
```
原有流程:
ReactAgent → @tool装饰函数 → MCP客户端

优化后流程:
ReactAgent → EnhancedSkill.to_langchain_tool() → 同原有@tool函数 → MCP客户端
         （仅描述优化）        （调用逻辑不变）
```

### 测试方法

2. **对比观察**：记录优化前后 LLM 的思考步骤
3. **Token 估算**：通过思考步骤减少估算 Token 节约

### 成功标准
1. ✅ 原有功能100%正常
2. ✅ LLM 工具选择准确率显著提升（主观评估）
3. ✅ 代码变更量最小，易于维护
4. ✅ 随时可回退，风险可控

## ⚠️ 风险与缓解

| 风险 | 可能性 | 缓解措施 |
|------|--------|----------|
| **描述过度复杂** | 中 | 保持描述简洁，只增加必要信息 |
| **转换逻辑错误** | 低 | 保持 func 直接引用原函数 |
| **性能影响** | 极低 | 仅初始化时转换一次，运行时无差异 |
| **兼容性问题** | 低 | 保留原 @tool 函数，随时切换 |

## 🎯 总结

**核心价值**：通过**最小代码变更**，实现**最大描述优化**，帮助 LLM 更智能地选择工具，从而减少思考步骤、节约 Token。

**关键优势**：
1. **简单易懂**：仅增强描述，不改变架构
2. **效果明显**：清晰的描述直接帮助 LLM 决策
3. **风险极低**：随时可回退到原有模式
4. **扩展性强**：未来可逐步优化更多工具
